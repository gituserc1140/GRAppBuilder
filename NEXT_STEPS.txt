GRAppBuilder - Next Steps

Current state
- Streamlit app deployed from this repo.
- GitHub PAT authentication is implemented.
- Repo creation is implemented.
- Starter template support is implemented.
- Current template: OpenWeather Streamlit app.
- Generated repo includes starter files, requirements, README, and .streamlit/secrets.toml.example.

Current user flow
1. User opens GRAppBuilder.
2. User authenticates with GitHub PAT.
3. User creates a repo.
4. User selects a starter template.
5. App seeds the repo with starter files through the GitHub API.
6. User adds secrets in Streamlit Community Cloud and deploys the generated app.

Important notes
- Keep secrets out of Git.
- Streamlit Cloud secrets should be used for GITHUB_PAT and OPENWEATHER_API_KEY.
- GetWeather1.txt contained a plain API key and is now gitignored.
- That OpenWeather key should be rotated/regenerated if it is active.

AI builder architecture notes
- The model should not directly control GitHub.
- Safer flow:
  1. User prompt
  2. AI returns structured output
  3. App validates output
  4. App writes files to GitHub via API
- This can work with OpenAI or Cohere.
- Main concerns are cost, output quality, and validation.

Latest roadmap update (June 2026)

Product positioning
- GRAppBuilder should be positioned as an app-builder helper with human-in-the-loop iteration.
- One-shot generation is useful for scaffolding, but runtime fixes and refinement are expected.

Primary user journeys to implement
1. Quick prototype mode
   - Goal: fastest idea-to-working-demo flow.
   - Inputs: short prompt + optional API details.
   - Outputs: file bundle preview + optional one-click push.
   - UX: minimal controls, low friction, default settings.

2. Production hardening mode
   - Goal: improve reliability of generated apps before/after first deploy.
   - Add a hardening pass that requests:
     - Defensive error handling
     - Session state safety checks
     - Empty/null data guards
     - Better API failure and timeout handling
   - Include a smoke-test checklist the user can run quickly.

Suggested hardening checklist
1. Validate generated Python files compile cleanly.
2. Verify requirements.txt includes needed dependencies only.
3. Check secrets usage (no hardcoded keys or tokens).
4. Verify API calls have timeout + error messages.
5. Confirm empty state handling for forms/lists/tables.
6. Test create/update/delete flow for generated app features.
7. Add basic README run/deploy steps if missing.

Near-term implementation order
1. Add explicit mode selector: Quick prototype vs Production hardening.
2. Implement AI hardening pass prompt + apply changes to generated bundle.
3. Add post-generation smoke-test checklist UI.
4. Add per-file edit-before-push capability.
5. Add optional issue-fix loop: paste runtime error -> targeted patch generation.
